# 代码改造任务清单

## 1. 改造目标

这份清单只讨论**代码怎么改**，不展开论文写作。

总原则是：

1. **先保留当前基线可运行**，不要直接在 `WGDT` 上硬改。
2. **先替换最脆弱的对齐与拒识逻辑**，也就是 `DANN + batch权重 + radius`。
3. **先做 UOT v1 核心版**，确认方法学收益。
4. **最后再升级编码器到 Mamba-MAE**，避免一次性把变量加太多。

建议采用“**平行新模型**”策略，而不是在原文件里大面积覆写。

---

## 2. 当前代码里哪些地方先不要动

在第一轮改造中，以下部分建议暂时不动或只做最小改动：

- `utils/dataLoader.py`
- `utils/Trainer.py`
- `utils/splitData.py`
- `model/DCRN.py`
- `model/WGDT.py`

原因：

- 这些文件已经构成了一个能跑的最小训练闭环
- 第一轮目标是验证“对齐/拒识机制”而不是“编码器升级”
- 如果一开始就同时改 backbone、训练流程、损失函数，后面很难定位问题来源

## 4. 第二阶段：实现 Prototype Memory，替换固定 Anchor

这是第一优先级模块。

当前 `model/Anchor.py` 的核心问题是：

- anchor 是固定 one-hot 几何结构
- 它不是从真实特征空间中学习出来的
- 它和域偏移场景下的类语义原型不一致

### 4.1 新增文件：`model/Prototype.py`

建议新增一个类：`PrototypeMemory`

### 4.2 `PrototypeMemory` 建议接口

#### 初始化

```python
PrototypeMemory(
    num_classes: int,
    feat_dim: int,
    momentum: float = 0.99,
    normalize: bool = True
)
```

#### 关键成员

- `self.prototypes`: `shape = [C, D]`
- `self.initialized`: 标记哪些类已经初始化

#### 建议方法

1. `reset()`
   - 清空原型状态

2. `update(features, labels)`
   - 输入源域特征和标签
   - 用 EMA 更新每个类原型

3. `get_prototypes()`
   - 返回 `[C, D]` 原型矩阵

4. `compute_logits(features)`
   - 返回每个特征到各类原型的相似度或负距离

5. `predict(features)`
   - 返回预测类别与已知类分数

### 4.3 训练阶段接入方式

在 `model/UOT_OSDA.py` 中：

- 源域前向输出 `features_s`
- 根据 `labels_s` 调用 `prototype_memory.update(features_s, labels_s)`
- 用 `compute_logits(features_s)` 做源域分类监督

### 4.4 这一阶段的校验点

至少验证：

1. 原型是否会更新
2. 分类 logits 是否维度正确
3. 源域分类 loss 是否能下降

### 4.5 需要补上的避坑点：Prototype Cold Start

这是非常关键的一点。

如果在训练最开始、编码器特征还很混乱时就直接用 EMA 更新原型，原型会被早期噪声污染，后面很难纠正。

建议在 `model/Prototype.py` 中显式加入 warm-up 机制：

#### 方案 A：训练预热后再启用 EMA

- 前 `warmup_epochs` 轮只训练源域分类
- 这期间原型不做 EMA，或者只用当前 batch 均值覆盖更新
- `warmup_epochs` 之后再切换为 EMA 更新

建议参数：

- `warmup_epochs = 5 ~ 10`
- `momentum` 从 `0.0` 或 `0.5` 逐步升到 `0.99`

#### 方案 B：利用预训练结果初始化原型

如果保留原仓库的 `pre_train` 逻辑，那么在预训练结束后：

1. 跑一遍全量源域训练集
2. 提取所有源域特征
3. 对每个类计算全局均值
4. 用这个均值初始化 `self.prototypes`

然后再进入主训练的 EMA 更新阶段。

#### 额外建议

为了减少尺度漂移，建议：

- 对 `features` 做 `L2 normalize`
- 对 `prototypes` 也做 `L2 normalize`

这样能显著减小后续 UOT 代价矩阵的尺度波动。

### 5.4 v1 版本复杂度控制建议

第一版不要直接做全局样本到样本 OT，建议做：

- **源端：类原型**
- **目标端：当前 batch 特征**

也就是 prototype-to-batch UOT。

先跑通，再考虑升级成：

- prototype-to-cluster UOT
- memory-bank UOT
- graph-regularized UOT

### 5.5 需要补上的避坑点：Sinkhorn 显存与梯度流向

这是实现时必须提前规定的工程策略。

如果手写 Sinkhorn 迭代，并让每一步都参与自动求导，那么：

- 计算图会非常大
- 显存占用会迅速上升
- 反向传播极慢
- 很容易出现数值下溢和 `NaN`

因此建议：

#### v1 默认策略：阻断 OT 求解器梯度

在第一版中，建议把 `transport_plan` 当作 detached 的软分配结果：

```python
with torch.no_grad():
    transport_plan = sinkhorn_unbalanced(...)
```

然后用这个 `transport_plan` 去构造最终 loss，例如：

- 原型加权匹配 loss
- soft pseudo-label loss
- 已知类分配置信度

这样做更像一个稳定的“E-step / assignment-step”。

#### v1.5 才考虑可导 OT

如果后面实验表明可导 OT 很有必要，再考虑：

- log-domain Sinkhorn
- truncated backprop
- 或调用更稳定的 OT 库

但这一步不要放进第一轮核心实现里。

#### 额外工程建议

- Sinkhorn 实现优先采用 log-domain 版本
- 先用 `prototype-to-batch`，不要直接全样本 OT
- 统计 `transport_plan` 是否出现全 0、全 `dustbin`、`NaN`

### 5.6 在 `model/UOT_OSDA.py` 中的接入方式

训练时：

1. 编码源域和目标域特征
2. 更新源域原型
3. 计算源域分类 loss
4. 用 `prototype` 和 `target feature` 构造 UOT 代价矩阵
5. 求得 `transport_plan`
6. 计算 `uot_loss`

### 5.7 这一阶段的交付标准

至少实现：

1. 不再依赖域判别器
2. 不再使用 batch min-max 权重
3. 可以输出每个目标样本的运输分配信息

## 6. 第四阶段：加入 Dustbin 拒识通道

这一阶段的目标，是正式替代 `Radius`。

### 6.1 当前需要被替换的旧逻辑

以下逻辑最终要从新模型中移除：

- `model/Radius.py`
- `model/WGDT.py` 中 `gamma -> radius -> unknown`

### 6.2 Dustbin 的最小实现

在 `model/UOT.py` 中新增：

- 一个额外源端行表示 `dustbin`
- 所有目标样本都可以把一部分质量流向 `dustbin`

最小版本先不做复杂正则，只输出：

- `dustbin_mass_per_target`

这可以直接作为 unknown score 的候选。

### 6.3 建议在 `model/UOT_OSDA.py` 中新增的推理逻辑

推理时对每个目标样本输出：

1. `class_scores`
2. `dustbin_score`
3. `prediction_known`
4. `prediction_final`

推荐最小决策规则：

```python
if dustbin_score > threshold:
    prediction = unknown
else:
    prediction = argmax(class_scores)
```

### 6.4 阈值策略

第一版不要上 EVT，先做稳定方案：

- 在源域已知类分数上做 percentile 阈值
- 或在训练期间做 EMA 统计

后续再升级成：

- EVT / Weibull
- 温度校准
- 双分数融合校准

### 6.5 需要补上的避坑点：Dynamic Dustbin Cost 不能退回纯 Batch Heuristic

这个建议方向是对的，但需要再收紧一下。

如果 `dustbin_cost` 完全由“当前 batch 的距离中位数或 80 分位数”决定，虽然比固定常数更灵活，但它会重新引入 batch 依赖，和我们前面要规避的 `batch min-max` 问题有相似风险。

因此更推荐：

#### 推荐做法：半全局自适应 `dustbin_cost`

定义：

- 当前 batch 的原型距离统计量：`q_batch`
- 运行中的 EMA 统计量：`q_ema`

然后使用：

```python
dustbin_cost = beta * q_ema + (1 - beta) * q_batch
```

其中 `beta` 建议较大，例如：

- `0.9 ~ 0.99`

这样既保留自适应性，又不会完全受单个 batch 支配。

#### 比纯 batch 方案更稳的两个变体

1. 用 **源域已知类到原型的距离分位数** 初始化 `dustbin_cost`
2. 用 **memory bank / recent queue** 估计目标域距离分布，而不是只看当前 batch

#### 后续增强版

如果后面发现类间尺度差异很大，可以再升级成：

- per-class `dustbin_cost`
- cluster-aware `dustbin_cost`

### 6.6 这一阶段的交付标准

至少做到：

1. 新模型能输出 unknown score
2. 推理不再依赖 `radius`
3. 单样本和小 batch 推理不崩
4. `dustbin_cost` 不依赖纯当前 batch 的瞬时极值

### 8.3 需要补上的避坑点：H-score 是北极星指标，但不能无条件拿来做早停

这个提醒非常重要，但要分两层看。

#### 第一层：指标层面

我同意，`HOS/H-score` 应该成为最核心汇报指标，因为它最能反映：

- 已知类识别能力
- 未知类拒识能力

之间的平衡。

所以：

- 论文主表优先看 `HOS`
- 调参时优先关注 `HOS`

#### 第二层：训练选择层面

但在**严格无监督目标域适应**设定下，不能直接用目标域真值 `H-score` 做 early stopping，否则会引入 oracle 选择偏差。

因此建议分开：

##### 方案 A：研究阶段的 oracle 上界

- 可以额外报告“best target HOS”
- 但必须明确标注为 `oracle selection`

##### 方案 B：正式无监督选择策略

训练时不用目标真值做早停，而用代理指标，例如：

- 源域验证分类性能
- 目标域 `dustbin ratio` 稳定性
- 目标域 unknown score 分布稳定性
- UOT 运输熵或运输稀疏性

也就是说：

> `HOS` 是最终北极星汇报指标，但不应默认成为严格无监督训练时的 checkpoint 选择指标。

### 8.4 对 `Trainer.py` 的额外改造建议

当前 `utils/Trainer.py` 只负责训练和测试，没有“最佳模型保存”逻辑。

因此后面如果要支持 checkpoint 选择，建议新增：

- `best_metric_name`
- `best_metric_value`
- `save_best_checkpoint()`

但第一版先不要把逻辑做太重，可以先：

1. 每个 epoch 保存一份轻量日志
2. 训练结束后统一选最优 epoch
3. 后续再加真正的 best-checkpoint 机制

### 8.5 可视化建议

建议新增一个简单可视化文件，比如：

- `utils/visual_uot.py`

1. `unknown score` 直方图

3. `feature` 的 t-SNE/UMAP

第一版即使先不写，也要把数据保存下来，别把调试信息丢掉。

## 13. 每一阶段的验收标准

### 验收 A：骨架接通

## 14. 风险与回退策略

### 风险 1：UOT 求解数值不稳

回退方案：

- 先做 entropic regularized balanced OT
- 再逐步放开到 UOT

### 风险 2：Dustbin 吸走过多已知样本

回退方案：

- 提高 `dustbin_cost` 的全局基准
- 限制 dustbin 总流量
- 只对低置信目标样本开放 dustbin
- 用 EMA 分位数替代纯 batch 动态代价

### 风险 3：编码器升级后训练过重

回退方案：

- 先保留 DCRN + MAE decoder
- 再换成 Mamba encoder

### 风险 4：一次改动太多难以调试

回退方案：

- 严格按 `v1 -> v2` 路线推进
- 不在第一轮同时引入 MAE、UOT、graph regularization、EVT
