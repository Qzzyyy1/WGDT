## 3. 首次跑库检查清单

第一次在服务器上跑 `UOT_OSDA`，不要先盯最终精度，先看训练链路是否正常。

### 3.1 必看日志项

- `loss_cls`
- `loss_uot`
- `source_oa`
- `running_threshold`
- `target_dustbin_ratio`
- `[ORACLE Bounds] Target H-Score`
- `dustbin_cost`

### 3.2 判断标准

#### 1）`loss_cls`

- 应为有限值
- 不应出现 `nan/inf`
- 应有下降趋势，或者至少逐渐稳定

#### 2）`loss_uot`

- 应为有限值
- 若很快爆炸或变成 `nan`，优先调 `uot_epsilon`

#### 3）`source_oa`

- 源域分类应该逐渐变好
- 如果始终很差，优先怀疑：
  - `prototype_warmup_epochs`
  - `pre_train`
  - 原型更新太早

#### 4）`running_threshold`

- 应缓慢变化
- 不应每个 epoch 大幅跳动

#### 5）`target_dustbin_ratio`

这是 `UOT_OSDA` 最关键的健康指标。

- 如果接近 `0%`：垃圾桶饿死，未知拒识失效
- 如果接近 `100%`：垃圾桶吸走一切，全部预测为未知
- 更合理的早期范围通常在 `10% ~ 60%`

#### 6）`[ORACLE Bounds] Target H-Score`

- 这是上界观察指标
- 仅用于判断 `UOT` 对齐是否在起正向作用
- 不能当作正式无监督结果进行早停选择

## 4. 常见异常与处理建议

### 4.1 `loss_uot = nan / inf`

优先尝试：

1. 调大 `uot_epsilon`
   - `0.05 -> 0.1 -> 0.2`
2. 减小 `batch`
3. 观察 `dustbin_cost` 是否极端

### 4.2 `target_dustbin_ratio ≈ 0`

说明垃圾桶基本没在工作。

优先尝试：

1. 降低 `dustbin_quantile`
   - 例如 `0.9 -> 0.85 -> 0.8`
2. 降低 `threshold_quantile`

### 4.3 `target_dustbin_ratio ≈ 1`

说明垃圾桶吸走了一切。

优先尝试：

1. 提高 `dustbin_quantile`
2. 降低 `uot_loss_weight`
3. 适当增大 `prototype_warmup_epochs`

### 4.4 `source_oa` 起不来

优先尝试：

1. 打开 `pre_train=True`
2. 增大 `pre_train_epochs`
3. 增大 `prototype_warmup_epochs`

### 4.5 `oracle_hos` 一直不涨

优先调整：

1. `uot_epsilon`
2. `dustbin_quantile`
3. `threshold_quantile`
4. `uot_loss_weight`

不要一开始就同时改很多参数。

## 7. 不同数据集的首次推荐命令

这里的命令适合作为“第一次正式上服务器”的初始版本，优先目的是稳定地跑出第一批日志。

## 7.1 `PaviaU_7gt -> PaviaC_OS`

### 推荐理由

- 相对更适合作为通路验证数据集
- 建议先用它确认 `UOT_OSDA` 链路没问题

### 推荐命令

```bash
python main.py \
  --model_name UOT_OSDA \
  --log_name UOT_OSDA \
  --device 0 \
  --source_dataset PaviaU_7gt \
  --target_dataset PaviaC_OS \
  --pre_train True \
  --pre_train_epochs 10 \
  --epochs 150
```

### 当前阶段推荐配置（已通过短程实验验证）

下面这组参数是当前 `PaviaU_7gt -> PaviaC_OS` 的第一版有效配置，已经在 `5 epoch` 短程实验中验证出明显优于早期版本：

- `dustbin_mass_prior = 0.2`
- `dustbin_quantile = 0.75`
- `unknown_score_alpha = 0.8`
- `threshold_quantile = 0.85`

对应现阶段推荐的中程验证命令为：

```bash
python main.py \
  --model_name UOT_OSDA \
  --log_name UOT_OSDA \
  --device 0 \
  --source_dataset PaviaU_7gt \
  --target_dataset PaviaC_OS \
  --pre_train True \
  --pre_train_epochs 10 \
  --epochs 30 \
  --dustbin_mass_prior 0.2 \
  --dustbin_quantile 0.75 \
  --unknown_score_alpha 0.8 \
  --threshold_quantile 0.85
```

### 当前结论（截至本轮）

- `mass` 主导的 unknown score 明显优于早期均衡融合版本
- `dustbin_mass_prior = 0.2` 比默认 `0.05` 更符合当前 `PaviaC_OS` 的开放程度
- `threshold_quantile = 0.85` 比默认 `0.95` 更合理
- 当前短程实验已经把 `HOS` 从接近 `0` 推到约 `0.32`，说明方向有效，值得进入 `30 epoch` 中程验证

### 当前 30 epoch 中程结果（已验证）

使用配置：

- `dustbin_mass_prior = 0.2`
- `dustbin_quantile = 0.75`
- `unknown_score_alpha = 0.8`
- `threshold_quantile = 0.85`

得到的阶段性结果：

- `oa = 0.4073`
- `aa_known = 0.3799`
- `unknown = 0.4553`
- `hos = 0.4142`
- `auroc_unknown = 0.5279`
- `oracle_hos` 峰值约 `42.38%`

当前判断：

- 这组配置已经是 `Pavia` 当前第一版有效配置
- 曲线在 `10~15 epoch` 后开始进入平台区
- 不建议原封不动继续直接扩到 `150 epoch`
- 更合理的动作是做 `1~2` 组邻域微调，再确定最终 `Pavia` 配置

## 8. 推荐的实验推进顺序

建议不要四个数据集一起盲跑，推荐顺序如下：

1. `PaviaU_7gt -> PaviaC_OS`
2. `Yancheng_ZY -> Yancheng_GF`
3. `Houston13_7gt -> Houston18_OS`
4. `HyRank_source -> HyRank_target`

原因：

- 先用相对容易的数据集验证链路
- 再用困难数据集暴露 `UOT` 是否稳

## 12. 当前建议结论

如果现在开始第一轮正式实验，我建议这样做：

1. 先跑 `PaviaC_OS` 的 5 epoch 短程测试
2. 没问题后跑 `PaviaC_OS` 全量
3. 再跑 `Houston18_OS` 5 epoch 短程测试
4. 如果 `Houston` 稳，再扩到 `HyRank` 和 `Yancheng`

这套顺序最稳，也最容易尽快得到第一批可分析的日志。
